How This Terraform Load Balancer Actually Works

Think of this setup as a chain. A user enters the public IP, Google receives the request, then the load balancer decides where to send it, and finally one backend VM answers with a web page. Every Terraform resource in main.tf exists to support one part of that path.

The Big Idea

A load balancer is not just one resource. It is a group of connected pieces. Beginners often expect one Terraform block called "load balancer", but in Google Cloud the load balancer is assembled from several resources that each do one job.

  • The front door is the global IP and forwarding rule.
  • The traffic brain is the proxy and URL map.
  • The backend layer is the backend service, health check, instance group, and VM.
  • The network foundation is the VPC, subnet, and firewall rule.

Simple Analogy

  • Global IP: the public street address
  • Forwarding rule: the front gate
  • HTTP proxy: the receptionist
  • URL map: the routing instructions
  • Backend service: the team manager
  • Health check: the attendance check
  • Instance group: the team roster
  • VM: the actual worker doing the job

Traffic Flow

Read this from top to bottom. This is the order a real HTTP request follows.

Step 1

User visits the public IP

The user sends an HTTP request to the public IP that Terraform reserved. In this project that IP comes from google_compute_global_address.default.

Step 2

Forwarding rule catches the request

google_compute_global_forwarding_rule.default listens on port 80 and knows which target should receive incoming traffic.

Step 3

HTTP proxy handles HTTP traffic

google_compute_target_http_proxy.default is the HTTP-aware component. It understands this is web traffic and consults the URL map.

Step 4

URL map chooses the backend service

google_compute_url_map.default says: "for this request, use the default backend service." In larger projects, this is where different paths like /api or /app can split.

Step 5

Backend service selects healthy backends

google_compute_backend_service.default connects the load balancer to the backend group. It also refuses to send traffic to instances that fail the health check.

Step 6

Health check verifies port 80

google_compute_health_check.http repeatedly tests the backend on port 80. If the VM is not answering, the load balancer will keep it out of rotation.

Step 7

Instance group points to the backend VM

google_compute_instance_group.backend_group is the group the backend service talks to. Even though this example has only one VM, the load balancer still expects a group abstraction.

Step 8

The VM serves the page

google_compute_instance.backend runs a tiny Python web server on port 80 and returns the demo HTML page.

Public Entry: IP + Forwarding Rule Routing Layer: Proxy + URL Map Backend Layer: Backend Service + Health Check + Group + VM Network Layer: VPC + Subnet + Firewall

Resource-by-Resource Explanation

Foundation

VPC Network

The VPC is the private network that holds your backend infrastructure. It is like the private building where your servers live.

  • Purpose: create an isolated network for your resources
  • Why needed: the backend VM must live inside some network
google_compute_network.lb_vpc
Foundation

Subnetwork

The subnet is a smaller IP range inside the VPC. Your backend VM receives its internal IP from here.

  • Range in this demo: 10.10.1.0/24
  • Why needed: the VM needs an internal address
google_compute_subnetwork.lb_subnet
Security

Firewall Rule

A firewall decides what traffic is allowed into the backend network. In this project it allows Google health check systems to reach port 80 on the backend VM.

  • Purpose: allow the load balancer health check to reach the VM
  • Without it: the VM might be healthy, but Google could not prove it
google_compute_firewall.allow_http_to_backend
Compute

Backend VM

This is the machine that actually answers the request. In this example it is a single small Debian VM running a basic Python web server.

  • Purpose: serve the HTML page
  • Important point: the load balancer does not serve content itself, the VM does
google_compute_instance.backend
Grouping

Instance Group

The load balancer does not point directly at a raw VM. It points to an instance group. That group can hold one instance or many instances.

  • Purpose: provide a backend target for the backend service
  • Named port: tells Google that http means port 80
google_compute_instance_group.backend_group
Health

Health Check

The health check is how Google decides whether the backend should receive traffic. It repeatedly asks: "Is something listening on port 80 and responding?"

  • Purpose: keep traffic away from broken backends
  • Beginner lesson: healthy infrastructure means more than just "VM is running"
google_compute_health_check.http
Backend Logic

Backend Service

This is the layer that connects the load balancer to the backend group. It uses the health check and only forwards requests to healthy members.

  • Purpose: define which group serves traffic
  • Think of it as the manager between the front door and the worker machines
google_compute_backend_service.default
Routing

URL Map

The URL map decides where requests go based on hostname or path. In this simple example, all traffic goes to the same backend service.

  • Purpose: route requests
  • In bigger systems: this is where /api and /frontend can split
google_compute_url_map.default
Protocol Handling

HTTP Proxy

The target HTTP proxy receives HTTP traffic from the forwarding rule and uses the URL map to decide what to do.

  • Purpose: handle HTTP-specific routing behavior
  • Without it: the forwarding rule would have nowhere to send HTTP requests
google_compute_target_http_proxy.default
Public Entry

Global Address

This is the public IP address users hit from the internet. It is reserved separately so it can stay stable and be shown as an output.

  • Purpose: create the public address
  • User-facing result: this is what you test with curl
google_compute_global_address.default
Public Entry

Forwarding Rule

This is the front door. It says: "when someone reaches this IP on port 80, send the request to the HTTP proxy."

  • Purpose: connect public traffic to the load balancer chain
  • It is the first load balancer resource the internet touches
google_compute_global_forwarding_rule.default

What Beginners Usually Miss

1. A running VM does not automatically mean a healthy backend. The health check must succeed too.

2. The load balancer is assembled from multiple Terraform resources, and each one has a narrow responsibility.

3. The firewall is part of the load balancer story, because health checks need network access.

4. The backend service and instance group are the bridge between routing logic and actual servers.
