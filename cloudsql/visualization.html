How This Cloud SQL Terraform Example Works

Cloud SQL is a managed database service. That means Google runs the database server for you, and Terraform mainly describes what database environment you want: the instance, the database inside it, and the user that can connect to it.

The Big Idea

In this example, Terraform creates the database stack in three layers:

  • Database server: the Cloud SQL instance
  • Database container: the logical database inside the instance
  • Access identity: the database user and password

Simple Analogy

  • Instance: the apartment building
  • Database: one apartment inside the building
  • User: the person who has a key

Creation Flow

Step 1

Create the Cloud SQL instance

Terraform first creates the managed database server itself. This is google_sql_database_instance.main.

Step 2

Create the logical database

Inside that instance, Terraform creates a database named for your application. This is google_sql_database.app.

Step 3

Create the application user

Terraform then creates a database user with a password. This is google_sql_user.app.

Resource-by-Resource Explanation

Managed Database Server

Cloud SQL Instance

This is the real database server managed by Google. You choose its engine version, region, and machine size.

  • Purpose: create the database server itself
  • In this example: MySQL 8 with a small demo tier
google_sql_database_instance.main
Logical Database

Database

A Cloud SQL instance can hold multiple databases. This resource creates one of them for your app.

  • Purpose: create the schema container your app will use
  • Example name: appdb
google_sql_database.app
Access Control

Database User

The user is what your application logs in with. It needs a username and password.

  • Purpose: allow your app to connect to the database
  • Important: keep the password local and secret
google_sql_user.app
Useful Outputs

Connection Information

After apply, Terraform can show values such as the instance name, connection name, and public IP to help you connect from other services.

  • cloudsql_instance_name
  • cloudsql_connection_name
  • cloudsql_public_ip

What Beginners Usually Miss

1. Cloud SQL is managed, so you do not create a VM yourself.

2. The instance is not the same thing as the database inside it.

3. A user is separate from the database and can be managed independently.

4. Passwords should stay in local-only files like terraform.tfvars.
